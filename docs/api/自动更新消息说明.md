# 自动更新消息说明

本文档说明服务端需要推送什么消息，客户端才会弹出更新公告、显示下载进度，并在下载完成后自动启动新版本。

## 1. 发送方式

服务端继续沿用现有的 `server-message` 推送逻辑即可，不需要单独新增一套通信。

客户端会识别下面两类写法：

1. 直接把更新消息作为服务器推送消息发出
2. 把更新消息放在 `data`、`payload`、`announcement` 或 `update` 里

## 2. 推荐消息类型

建议服务端把更新消息的 `type` 或 `message_type` 设为以下任意一种：

- `app_update`
- `update`
- `software_update`
- `upgrade`

只要带了 `version` 和 `downloadUrl`，客户端也会尝试识别为更新消息。

## 3. 必填字段

更新消息至少要有这两个字段：

- `version`：目标版本号
- `downloadUrl`：更新包下载地址

示例：

```json
{
  "type": "update",
  "version": "2.5.2",
  "downloadUrl": "https://example.com/releases/ai-free-2.5.2.zip"
}
```

## 4. 可选字段

这些字段不是必须，但建议提供：

- `title`：更新标题
- `content`：更新说明或公告正文
- `fileName`：下载保存时的文件名
- `entryFile`：解压后要启动的文件名
- `force`：是否强制更新

示例：

```json
{
  "type": "app_update",
  "title": "发现新版本",
  "content": "修复了下载流程问题，建议立即更新。",
  "version": "2.5.2",
  "downloadUrl": "https://example.com/releases/ai-free-2.5.2.zip",
  "fileName": "ai-free-2.5.2.zip",
  "entryFile": "AI-FREE.exe",
  "force": false
}
```

## 5. 客户端行为

客户端收到更新消息后会：

1. 读取当前软件版本
2. 和服务端传来的 `version` 比较
3. 如果当前版本已经不低于目标版本，则不提示更新，只做忽略处理
4. 如果当前版本低于目标版本，则弹出更新公告
5. 用户点击 `立即更新` 后，客户端自动下载 `downloadUrl`
6. 下载过程中，软件底部会显示进度条
7. 下载完成后自动解压
8. 自动启动解压后的目标文件
9. 退出当前软件

## 6. 更新包要求

### 6.1 推荐格式

优先推荐：

- `.zip`
- `.tar.gz`
- `.tgz`
- 单个 `.exe`

### 6.2 推荐目录结构

如果是压缩包，建议把主程序放在压缩包根目录，或者明确写 `entryFile`。

例如：

```text
ai-free-2.5.2.zip
  ├─ AI-FREE.exe
  ├─ resources/
  └─ ...
```

或者：

```text
ai-free-2.5.2.zip
  ├─ release/
  │  ├─ AI-FREE.exe
  │  └─ ...
```

然后在消息里指定：

```json
{
  "version": "2.5.2",
  "downloadUrl": "https://example.com/releases/ai-free-2.5.2.zip",
  "entryFile": "release/AI-FREE.exe"
}
```

### 6.3 单文件程序

如果服务端直接下发的是一个可执行文件，也可以直接启动。

例如：

```json
{
  "type": "update",
  "version": "2.5.2",
  "downloadUrl": "https://example.com/releases/AI-FREE.exe"
}
```

## 7. 强制更新

如果要强制更新，可以把 `force` 设为 `true`。

```json
{
  "type": "update",
  "version": "2.5.2",
  "downloadUrl": "https://example.com/releases/ai-free-2.5.2.zip",
  "force": true
}
```

前端当前仍会显示 `立即更新` 和 `稍后更新` 两个按钮，`force=true` 主要用于服务端表达更新优先级。若后续需要真正禁止跳过，可以再扩展前端逻辑。

## 8. 服务端示例

### 8.1 普通更新公告

```json
{
  "type": "announcement",
  "message_type": "update",
  "title": "版本更新",
  "content": "2.5.2 已发布，请尽快更新。",
  "version": "2.5.2",
  "downloadUrl": "https://example.com/releases/ai-free-2.5.2.zip",
  "entryFile": "AI-FREE.exe"
}
```

### 8.2 直接更新消息

```json
{
  "type": "update",
  "title": "发现新版本",
  "content": "修复若干问题。",
  "version": "2.5.2",
  "downloadUrl": "https://example.com/releases/ai-free-2.5.2.zip",
  "fileName": "ai-free-2.5.2.zip",
  "entryFile": "AI-FREE.exe"
}
```

### 8.3 强制更新消息

```json
{
  "type": "app_update",
  "title": "必须更新",
  "content": "当前版本已停止服务，请立即更新。",
  "version": "2.5.2",
  "downloadUrl": "https://example.com/releases/ai-free-2.5.2.zip",
  "force": true
}
```

## 9. 不要这样发

以下情况容易导致客户端无法正确识别更新：

- 只有公告文本，没有 `version`
- 只有 `version`，没有 `downloadUrl`
- 下载地址不是可直接访问的文件地址
- 压缩包里没有可执行文件，且没有提供 `entryFile`
- 把更新内容伪装成 `shutdown` 或停用消息

## 10. 兼容说明

客户端当前会同时兼容以下字段名：

- `version`
- `latest_version`
- `latestVersion`
- `new_version`
- `newVersion`
- `target_version`
- `targetVersion`
- `app_version`
- `appVersion`

下载地址兼容：

- `downloadUrl`
- `download_url`
- `packageUrl`
- `package_url`
- `fileUrl`
- `file_url`
- `url`
- `link`

## 11. 建议的最小实现

服务端最少只需要保证每条更新消息包含：

- `type: "update"`
- `version`
- `downloadUrl`

如果要提升展示效果，再补：

- `title`
- `content`
- `entryFile`
- `fileName`
