# AI-FREE 客户端彻底整理与可维护性改造计划

> 实施状态（2026-07-19）：阶段 0–5 与最终门禁已完成。守卫基线为零，334 项测试、覆盖率、真实 Electron/Chromium、Windows 构建和 packaged runtime 均通过。逐项证据见 [PROGRESS.md](PROGRESS.md)，功能覆盖见 [stage0/feature-matrix.md](stage0/feature-matrix.md)。

## 1. 背景与改造目标

当前客户端已经形成 Electron 主进程、Chromium Runtime、侧边栏和扩展等基本边界，但仍存在以下维护问题：

- 多个核心文件同时承担状态、业务、IO、IPC 和界面职责，单文件达到 1,100–2,900 行。
- 渲染层依赖 HTML 脚本加载顺序和 `window.*` 全局对象，调用关系不够明确。
- preload 暴露任意 IPC 通道，通道、输入、输出和错误格式没有集中契约。
- 大量测试通过读取源码并匹配文本验证，拆文件后容易产生与真实功能无关的失败。
- 缺少统一的 lint、类型检查、覆盖率、依赖边界和模块功能完整性门禁。

本次整改的目标不只是“代码能够运行”，而是让每个模块都具备明确职责、稳定接口、完整功能清单、可执行行为测试和可追踪诊断信息。任何模块只有在功能正确性、完整性和兼容性均验证通过后，才能判定整改完成。

总体原则：

- 保持现有 UI、用户流程、HTTP 协议、打包结果和本地数据兼容。
- 先建立测试和架构护栏，再逐个业务域替换，禁止一次性无保护重写。
- 每阶段均保持可启动、可测试、可打包、可回滚。
- 测试必须主动验证功能，不得只验证“不报错”“代码存在”或“源码包含某段文本”。
- 发现功能缺口时先补充可复现测试，再修正实现和功能清单。

## 2. 范围与默认边界

本轮包含：

- `src/app` 下 Electron 主进程、preload、应用外壳、侧边栏和共享代码。
- 自研 `browser_automation` 扩展。
- 对应的测试、开发脚本、构建脚本和架构文档。

本轮不直接重写：

- `backserve` 后端；只验证现有 HTTP 契约兼容。
- Chromium fork 和原生宿主内部实现；只验证其公开接口和运行行为。
- `remove_watermark`、`transform` 等第三方扩展资产；禁止格式化或结构调整。

默认冻结产品行为。整改过程中只修复有明确复现步骤和自动化测试的缺陷，不顺带改变交互或业务规则。

## 3. 目标架构

### 3.1 目录与依赖方向

保留 Electron 进程边界，在每个进程内按业务域组织：

- `main/composition`：唯一应用装配入口，只负责创建依赖和生命周期注册。
- `main/platform`：Electron、文件系统、子进程、网络、路径和日志适配器。
- `main/features/{ai-chat,browser,network,account,extensions,updates}`：各域的 service、repository、IPC adapter 和类型。
- `renderer/features`、`sidebar/features`：各域的状态、视图、事件绑定和 API client。
- `contracts`：IPC 请求、响应、事件、错误码和共享类型。
- `shared`：不依赖 Electron、DOM、文件系统的纯函数。

强制依赖方向：

```text
UI / IPC adapter
        ↓
application service
        ↓
domain rule
        ↓
injected platform / repository interface
```

domain 不得反向依赖 Electron、DOM、具体文件路径或具体 HTTP 实现。

### 3.2 Electron 与 IPC 契约

- 建立集中 IPC 注册表，登记通道、方向、所属域、请求 schema、响应 schema 和权限。
- 禁止业务代码直接书写 IPC 通道字符串。
- 删除覆盖 `ipcMain.handle/on` 的去重补丁；重复注册应立即失败并指出来源。
- 每个 IPC 注册器返回 `dispose()`，窗口或应用退出时显式释放监听器。
- preload 最终只暴露按域分组的 `window.aiFree`，不再允许任意 `send/invoke/on`。
- 事件订阅统一返回取消订阅函数，并验证事件来源和 payload。
- 新接口统一返回：

```js
{ ok: true, data }
{ ok: false, error: { code, message, retryable, details? } }
```

- 迁移期间保留带白名单和废弃日志的 `window.electronAPI` 适配层；调用迁移完成后删除。
- HTTP 地址、字段和业务语义保持不变，在 repository 边界集中完成解析、校验和错误映射。

### 3.3 状态、存储与日志

- 用注入的 `AppContext` 代替业务性 `global.*`、跨模块可变单例和隐式闭包状态。
- 账号、聊天历史、Profile、许可证、扩展和代理配置统一通过 repository 访问。
- 继续读取现有路径和格式；新增数据版本时必须幂等迁移、先校验后写入并保留原始备份。
- 统一日志字段：`domain`、`operation`、`correlationId`、`profileId`、`channel`、`errorCode`。
- API Key、Cookie、卡密、设备凭证和临时 Token 必须脱敏，不进入普通日志或测试快照。

## 4. 模块功能测试完整性制度

### 4.1 功能覆盖矩阵

整改每个模块前，必须先建立模块功能覆盖矩阵。矩阵至少包含：

| 项目 | 必填内容 |
|---|---|
| 功能编号 | 稳定且可引用的编号，例如 `AI-CHAT-STOP-01` |
| 用户/调用方目标 | 该功能解决什么问题 |
| 入口 | UI、IPC、定时任务、启动恢复或外部事件 |
| 前置状态 | 登录、VIP、浏览器、网络、配置和持久化状态 |
| 正常流程 | 输入、状态变化、输出和可见结果 |
| 异常流程 | 网络、文件、进程、权限、数据和依赖失败 |
| 边界条件 | 空值、上限、重复操作、并发、取消和超时 |
| 状态恢复 | 重试、重启、退出、崩溃和旧数据恢复 |
| 自动化测试 | 对应测试文件和用例名称 |
| 人工/真实环境验收 | 自动化无法覆盖时的步骤与预期结果 |

矩阵中的功能项未全部具备测试或明确验收记录时，该模块不得标记完成。

### 4.2 每项功能的最低测试组合

每项业务功能至少验证：

1. 正常路径：合法输入产生完整、正确的业务结果。
2. 输入边界：空值、非法格式、最小值、最大值和超限值。
3. 权限与状态：未登录、非 VIP、未连接、处理中和退出中等状态。
4. 依赖失败：HTTP、文件系统、IPC、Chromium、Clash、扩展或子进程失败。
5. 并发与重复：连续点击、重复 IPC、并发请求、启动与停止竞争。
6. 取消与超时：用户中止、窗口关闭、应用退出、请求超时和迟到响应。
7. 持久化与恢复：保存后重启、旧格式读取、损坏数据、迁移失败和回退。
8. 副作用检查：无重复监听器、无孤儿进程、无错误删除、无敏感信息泄漏。
9. 跨模块联动：调用链上下游获得正确数据和状态，而非只验证单个函数返回。
10. 打包环境：涉及路径、preload、扩展、原生资源的功能必须在打包版复验。

禁止使用以下结果作为功能完成证明：

- 代码可以加载或没有抛异常。
- 页面上存在某个字符串、CSS 类或函数名。
- mock 永远返回成功时测试通过。
- 只覆盖主流程而未验证失败、恢复和重复操作。
- 单元测试通过但真实 Electron/Chromium 流程未执行。

### 4.3 测试层级

- 纯单元测试：domain 规则、schema、状态机、格式转换和错误映射。
- service 集成测试：使用受控的 Electron、HTTP、文件、时钟和子进程适配器替身，验证完整业务流程及副作用。
- IPC 契约测试：验证每个桥接方法的输入、输出、错误、来源权限和监听器释放。
- DOM 行为测试：真实触发输入、点击、选择、组合输入、渲染和销毁，不匹配实现源码。
- Electron 集成测试：启动真实窗口和 preload，验证主进程与渲染进程联动。
- Chromium/扩展验收：验证浏览器进程、Profile、代理、扩展注入和自动化桥。
- 打包版验收：在 `win-unpacked` 中复测资源路径、扩展混淆、原生宿主和更新链路。

源码文本测试只允许用于无法通过运行时验证的静态打包或安全约束，并必须在测试中说明原因。

### 4.4 缺陷闭环

- 每个发现的缺陷先增加能够稳定失败的回归测试。
- 修复后同时运行模块测试、直接上下游测试和全量测试。
- 缺陷若暴露功能矩阵缺项，必须补全同类边界场景，不能只增加一个特例断言。
- 修复记录必须包含根因、受影响功能编号、数据兼容影响和验证证据。

### 4.5 现有测试代码清理与重新组织

现有扁平 `test/*.test.js` 不作为必须保留的代码。整改时允许全部删除旧测试文件，但必须遵守“先迁移有效场景，后删除旧文件”：

- 仅匹配源码字符串、函数名、CSS 类或文件内容的测试直接废弃，不做等价搬运。
- 同时包含真实行为断言和源码断言的文件拆开处理：保留其功能意图，用新模块公开接口重写行为测试，删除源码断言和原文件。
- HTTP 传输、代理流量、数据迁移、扩展隔离、混淆执行、打包资源等真实行为或安全约束不得丢失；迁移为新的契约、集成或打包测试。
- 旧测试文件不建立 `legacy`、`archive` 或备份目录；功能矩阵确认新测试覆盖后直接删除，避免长期保留两套测试。
- `scripts/check-*`、`scripts/smoke-*`、`scripts/accept-*` 中的测试逻辑一并清点：自动化断言迁入测试目录，脚本目录只保留启动、构建以及需要人工操作的薄入口。

新的测试目录固定按测试职责和业务域组织：

```text
test/
├─ unit/
│  ├─ ai-chat/
│  ├─ browser/
│  ├─ network/
│  ├─ account/
│  ├─ extensions/
│  ├─ updates/
│  └─ shared/
├─ contract/
│  ├─ ipc/
│  ├─ http/
│  └─ storage/
├─ integration/
│  ├─ main/
│  ├─ renderer/
│  ├─ electron/
│  ├─ chromium/
│  └─ browser-automation/
├─ packaging/
│  ├─ windows/
│  ├─ assets/
│  └─ obfuscation/
├─ acceptance/                 # 真实环境验收定义和可自动执行部分
├─ fixtures/                   # 经过脱敏的最小输入数据
└─ helpers/                    # fake adapters、测试时钟、临时目录等
```

整理约束：

- 测试文件只测试一个业务能力；跨域场景放入 `integration`，不在单元测试中堆积多域源码。
- fixture 必须最小化、脱敏并说明对应 schema；禁止复制真实 Cookie、Key、Token、Profile 或大型运行资产。
- helper 只能提供测试基础设施，不能隐藏业务断言或重新实现被测业务规则。
- package scripts 按层提供 `test:unit`、`test:contract`、`test:integration`、`test:packaging`、`test:acceptance` 和全量 `test`。
- 整改完成后，`test/` 根目录不再散放测试文件，`scripts/` 中不再存在可由测试运行器执行的重复断言。

## 5. 分阶段实施

### 阶段 0：冻结现状与建立功能清单

- 记录启动、AI 对话、浏览器、网络魔法、账号、扩展、更新和退出流程。
- 清点全部 IPC、HTTP、本地文件、环境变量、全局状态和兼容逻辑。
- 为每个域建立功能覆盖矩阵，明确自动化与真实环境验收责任。
- 记录启动时间、侧边栏打开时间、内存、退出耗时等性能基线。

### 阶段 1：工程和测试护栏

- 引入 ESLint、格式检查、TypeScript `checkJs`、覆盖率和依赖边界检查。
- 新增 `npm run verify`，顺序执行格式、lint、typecheck、架构边界、测试和构建配置检查。
- 建立新的分层测试目录；将有效功能场景重写为真实模块、IPC 契约、DOM 行为和集成测试，然后删除全部旧的扁平测试文件。
- 清理测试性脚本：自动化断言迁入 `test/`，人工验收入口归入 `test/acceptance`，构建和启动脚本继续留在 `scripts/`。
- 设置渐进式门禁：生产 JS/TS 文件最终不超过 500 行，单函数不超过 80 行，复杂度不超过 15。
- 超限存量进入显式整改清单；新增代码不得扩大超限范围。

### 阶段 2：装配、类型和 IPC 基础设施

- 将 `bootstrap.js` 收缩为不超过 250 行的 composition root。
- 建立 `AppContext`、统一错误类型、集中路径解析和结构化日志。
- 建立 contracts、payload 运行时校验、窄化 preload 和可释放 IPC 注册器。
- 增加混合 JS/TS 构建链；主进程/preload 输出 CommonJS，renderer/sidebar 输出 ES modules。
- 开发启动、测试和 Windows 构建共用同一代码构建入口。

### 阶段 3：逐域整改主进程

每个域严格执行“功能矩阵 → 特征测试 → 提取规则 → service/repository → IPC adapter → 功能完整性验收 → 删除旧实现”：

1. AI：模型与配额、消息压缩、工具派发、中断、标题、浏览器选择、自动化卡片和聊天历史。
2. 浏览器：标签、Profile、历史、设置、Chromium runtime 和 AI 窗口工具。
3. 网络：Clash 资产、配置、订阅、Geo、延迟探测、进程状态机和代理切换。
4. 账号：认证、设备身份、许可证、VIP、存储和旧数据恢复。
5. 扩展与更新：扩展发现、兼容副本、注入、刷新，以及更新检测、下载、校验和安装。

完成一个域后必须执行该域全矩阵、相邻域联动、全量自动化和真实 Electron 冒烟测试，不能仅凭代码审查进入下一域。

### 阶段 4：渲染层与自研扩展

- 将 `ai-control.js` 拆为 store、API client、history、composer、message renderer、tool display、browser/card selector 和 bootstrap。
- 将标签栏、账号、VPN、浏览器设置改为显式模块依赖，消除业务性 `window.*`。
- 事件绑定统一管理，重复初始化和窗口销毁必须安全。
- CSS 按 tokens、layout 和业务组件拆分，测试验证行为与可访问状态，不匹配样式源码。
- 整理 `browser_automation` 的状态、缓存、消息和页面操作模块，保持 Manifest 和桥接协议兼容。

### 阶段 5：清理和正式切换

- 将稳定的 domain、application 和 contracts 模块逐步迁移 TypeScript。
- 删除兼容 preload、废弃 IPC、重复工具、无调用导出、裸全局变量和旧目录。
- 统一文件命名、错误码、日志标签和测试命名。
- 更新 `AGENTS.md`、架构索引、数据契约、功能矩阵和故障定位手册。
- 完成开发版、真实 Electron、`win-unpacked` 和安装包验收后再移除回滚实现。

## 6. 重点模块功能验收

### AI 控制

- 发送、停止、连续停止和迟到响应隔离。
- 消息窗口裁剪、摘要、工具调用与工具结果完整配对。
- 历史创建、保存、重命名、删除、本地/远端合并和损坏数据恢复。
- 多浏览器选择、默认选择、手动取消、连接刷新和名称关联。
- 自定义 API、配额、MCP 上限、VIP 门禁和自动化卡片。

### 浏览器

- 新建、关闭、重启、创建中取消和重复创建合并。
- Profile 保存/恢复/迁移/清理，运行中 Profile 不被误删。
- 标签、窗口、Chromium 进程和扩展连接状态一致。
- 网络环境、时区、语言、UA 和代理应用结果正确。
- 崩溃、应用退出和更新退出不产生孤儿进程或重复清理。

### 网络魔法

- 配置导入、订阅转换、Geo 本地化和缺失资源回退。
- 延迟探测的 TLS、超时、并发限制和结果排序。
- 启动/停止竞争、重复操作、进程异常退出和系统代理恢复。
- 流量计数、非代理流量排除、计数器回滚和无效快照。

### 账号与授权

- 登录、设备登录、登出、找回、卡密和设备身份校验。
- VIP 永久/有效/过期状态与所有主进程、渲染层门禁。
- 旧数据读取、错误数据隔离、迁移失败和敏感字段保护。

### 扩展与更新

- 内置/自定义扩展发现、导入、启停、删除、刷新和兼容副本。
- 临时 Token 仅写入运行时副本，源扩展和日志不得泄漏。
- 更新版本判断、公告、下载、校验、取消、失败恢复和安装退出。
- 开发版与打包版路径、混淆代码和原生资源行为一致。

## 7. 最终验收门槛

- 功能覆盖矩阵中的所有功能均有自动化测试或明确的真实环境验收记录。
- `npm run verify` 全部通过，已有有效回归场景不得因拆文件被删除。
- 每个整改域覆盖率至少达到 80% lines/statements/functions、70% branches；全体自有应用代码最终至少达到 75%/65%。
- 旧的扁平测试和源码文本断言已清除；所有保留场景位于约定的分层目录，并可通过对应 package script 独立执行。
- 无循环依赖、重复 IPC、任意通道 preload、未登记超大文件或新增业务性全局变量。
- 所有自有代码通过 lint 和类型检查，第三方资产明确排除。
- 本地账号、聊天历史、Profile、代理、许可证和扩展配置可无损原位读取。
