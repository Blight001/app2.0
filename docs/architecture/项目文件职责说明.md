## 1. 项目整体定位

这是一个基于 Electron 的桌面应用，核心特点是：

- 主进程负责窗口、标签页、IPC、许可证、代理和启动流程。
- 渲染进程负责顶部标签栏、侧边栏、弹窗和页面交互。
- `src/assets/extensions` 下包含多个浏览器扩展和内嵌工具模块。
- `appbuild/` 更偏向打包产物，不属于日常开发主源码。备份使用 `version/backup_*`（由脚本动态创建）。

## 2. 根目录与一级目录

### 2.1 入口与构建

- [`package.json`](../../package.json)：项目元数据、依赖、启动脚本、Electron Builder 打包配置入口。
- [`package-lock.json`](../../package-lock.json)：npm 锁定文件，固定依赖版本。
- [`scripts/windows/build.bat`](../../scripts/windows/build.bat)：Windows 下的一键打包脚本，调用 `npm run build:win` 生成安装包。
- [`scripts/windows/v-start.bat`](../../scripts/windows/v-start.bat)：Windows 下的一键启动脚本，执行 `npm start`。
- [`scripts/windows/v-debug.bat`](../../scripts/windows/v-debug.bat)：Windows 下唯一的 HTTP 调试启动脚本，预设本地服务器地址、平台名和调试环境变量。
- [`scripts/windows/backup.bat`](../../scripts/windows/backup.bat)：备份脚本，把 `src/`、`config/`、`docs/` 和关键脚本复制到 `version/backup_时间戳/`（version 目录由脚本动态创建）。
- [`scripts/run-electron.js`](../../scripts/run-electron.js)：包装启动 Electron CLI，清理 `ELECTRON_RUN_AS_NODE` 后再启动。
- 注册器桥接相关脚本已移除（历史功能）。

### 2.2 配置与说明

- [`config/platforms-config.json`](../../config/platforms-config.json)：平台配置总表，包含默认平台名、AppId、窗口图标、HTTP 卡密查询地址、目标 URL、允许平台列表和系统代理开关。
- [`config/self.yaml`](../../config/self.yaml)：平台/运行相关的 YAML 配置文件，主要用于构建或环境控制。
- [`README.md`](../../README.md)：英文项目说明，介绍功能、结构、构建方式。
- [`README_zh-CN.md`](../../README_zh-CN.md)：中文项目说明。
- [`docs/api/HTTP请求说明.md`](../api/HTTP请求说明.md)：HTTP 接口说明文档。
- [`docs/api/TCP请求说明.md`](../api/TCP请求说明.md)：TCP 接口说明文档。
- [`.gitignore`](../../.gitignore)：Git 忽略规则。
- `软件`：普通文件，不是源码目录；当前仓库里它更像是一个占位/标记文件，和程序逻辑无直接关系。
- `AI账号刷新时间客户端修改说明.md`：当前工作树未包含该文档；如果后续恢复，建议放入 `docs/operations/`。

### 2.3 产物与历史目录

- [`appbuild/`](../../appbuild/)：Electron Builder 输出目录，放安装包、可执行文件、压缩包、blockmap 等构建结果。
- `version/`（如果存在）：由 backup.bat 动态创建的备份目录，用于存放源码快照（当前工作树中可能不存在）。
- [`node_modules/`](../../node_modules/)：npm 安装依赖，不属于业务源码。
- `.git/`：Git 元数据目录，不属于业务源码。

## 3. 主进程源码 `src/app/main`

这部分是应用核心，负责启动、窗口、标签页、IPC、账户、许可证、代理、扩展注入和生命周期管理。

### 3.1 入口文件

- [`src/app/main/main.js`](../../src/app/main/main.js)：主入口，只负责引入 `entry/start-app.js`。
- [`src/app/main/entry/start-app.js`](../../src/app/main/entry/start-app.js)：启动入口，负责初始化运行日志、清理更新缓存并转交 `bootstrap.js` 启动主应用。
- [`src/app/main/composition/create-ui-bridge.js`](../../src/app/main/composition/create-ui-bridge.js)：UI 桥接组合层，负责侧边栏发送和控制台历史桥接。
- [`src/app/main/composition/startup-guards.js`](../../src/app/main/composition/startup-guards.js)：启动守卫组合层，负责单实例和 Windows AppUserModelId 处理。
- [`src/app/main/bootstrap.js`](../../src/app/main/bootstrap.js)：当前仍是主进程总装配入口，已从直接入口降级为可复用启动模块，且部分启动守卫/UI 桥接已迁出，后续应继续拆薄。
- [`src/app/main/preload.js`](../../src/app/main/preload.js)：预加载脚本，向渲染进程暴露 `electronAPI`、`env` 和若干桥接方法。

### 3.2 配置层

- [`src/app/main/config/index.js`](../../src/app/main/config/index.js)：集中管理运行时配置、核心目录路径、store 路径、默认目标 URL、TCP 配置等。

它的主要职责：

- 解析运行时 TCP 参数。
- 计算核心目录和 store 路径。
- 提供 `DREAM_TARGET_URL`、`getTcpConfig()`、`getServerBase()` 等基础配置接口。

### 3.3 运行时状态

- [`src/app/main/runtime/app-state.js`](../../src/app/main/runtime/app-state.js)：集中保存窗口、标签、侧栏、许可证、插件开关等主进程状态。
- [`src/app/main/runtime/app-console.js`](../../src/app/main/runtime/app-console.js)：收集控制台历史，并转发到侧边栏界面。
- [`src/app/main/runtime/license-cache.js`](../../src/app/main/runtime/license-cache.js)：维护许可证验证结果、绑定状态、平台信息和运行时配置缓存。

### 3.4 服务层

- [`src/app/main/services/app-shell.js`](../../src/app/main/services/app-shell.js)：应用窗口外壳和启动流程编排，负责创建主窗口、许可证窗口和页面加载。

## 8. 结构整理与分模块建议

这一节不是解释“现在是什么”，而是定义“后面应该怎么收敛”。目标只有四个：

- 降低入口文件复杂度，避免功能继续堆到 `bootstrap.js`。
- 建立明确模块边界，减少跨层调用和隐式状态共享。
- 把运行资源、文档、构建脚本和业务源码分开，降低认知噪音。
- 在不动 `src/assets/**` 的前提下，让宿主程序 `src/app/**` 更稳定、更可控。

### 8.1 当前维护成本高的核心原因

结合当前结构，主要问题不是“完全没分层”，而是“分层还不够硬”：

- [`src/app/main/bootstrap.js`](../../src/app/main/bootstrap.js) 仍然过重。
  它同时承担启动判断、依赖组装、状态管理、窗口编排、服务创建、生命周期注册，属于典型的组合根膨胀。
- [`src/app/main/lib/http-client.js`](../../src/app/main/lib/http-client.js) 和 [`src/app/main/lib/auth-cookie.js`](../../src/app/main/lib/auth-cookie.js) 是网络与鉴权边界。
  一个文件里混了协议常量、连接管理、降级策略、平台同步、URL 同步、账号刷新时间等多类逻辑；另一个文件里混了认证、Cookie、平台推断、目标地址、区域解析和 locale 注入。
- [`src/app/main/ipc/register.js`](../../src/app/main/ipc/register.js) 虽然已做拆分，但 IPC 还是以“通道分类”为主，不是以“领域能力”组织。
- 根目录同时放了构建脚本、接口文档、运行配置、备份脚本、产物目录和说明文档，长期会让新人判断不了“哪些文件会影响运行，哪些只是说明”。
- （原 registration 边界讨论已过时，相关代码已移除）

### 8.2 建议的目标目录结构

不改 `src/assets/**` 的前提下，建议把宿主程序收敛成下面这套结构：

```text
app2.1/
  docs/
    architecture/
    api/
    operations/
  scripts/
  config/
    platforms-config.json
    self.yaml
  src/
    app/
      main/
        entry/
        composition/
        domains/
          license/
          account/
          network/
          browser/
          （registration 相关代码已移除，不再作为 app2.1 当前目标）
          proxy/
        infrastructure/
          electron/
          persistence/
          http/
          tcp/
        ipc/
        runtime/
      renderer/
        pages/
          license/
          side-panel/
          shell/
        shared/
      views/
    assets/
      ...
  appbuild/
  version/（备份）
```

这个结构的重点不是名字，而是边界：

- `entry/`：只保留入口判断和启动转发。
- `composition/`：只做依赖装配，不写业务规则。
- `domains/`：按业务域拆分，而不是按工具名堆文件。
- `infrastructure/`：放 HTTP、TCP、文件存储、Electron 适配等基础设施实现。
- `renderer/pages/`：按页面拆，而不是所有控制器平铺。
- `docs/`：根目录里的接口说明、职责说明、客户端改动说明统一收进去。
- `config/`：把根目录运行配置集中，避免以后继续散落。

### 8.3 主进程推荐拆分方式

#### 1. 入口层

把当前入口压到最薄：

- [`src/app/main/main.js`](../../src/app/main/main.js)：只负责 `require('./entry/start-app')`。
- 新增 `src/app/main/entry/start-app.js`：承接运行日志、更新缓存清理和主应用启动转发。
- 后续如继续拆薄 `bootstrap.js`，可新增主应用组合层来集中创建服务实例并返回 `start()`。

这样做的价值：

- 入口只回答“启动哪个应用”。
- 组合层只回答“依赖怎么拼”。
- 业务层只回答“规则怎么跑”。

#### 2. 领域层

建议按“业务能力”拆，不按“utils/lib/services”继续横向扩张。

推荐先建立这些领域目录：

- `domains/license/`
  负责卡密验证、许可状态、解绑、许可证记录。
- `domains/account/`
  负责账号存储、账号切换、Cookie 导入、刷新时间。
- `domains/network/`
  负责服务端配置、教程地址、目标 URL、平台列表、连接状态。
- `domains/browser/`
  负责主窗口、标签页、侧边栏、BrowserView 生命周期。
- `domains/proxy/`
  负责 Clash Mini、系统代理、节点状态。
- registration 相关领域已从当前 app2.1 移除，不再新增对应目录。

每个领域内部统一成三类文件即可：

- `service.js`：对外暴露领域能力。
- `store.js`：领域状态与持久化映射。
- `mapper.js` 或 `adapter.js`：和外部协议/IPC/配置格式转换。

不要在一个领域里同时出现十几个散文件，否则只是换了目录，复杂度没有下降。

#### 3. 基础设施层

当前 `lib/` 和 `utils/` 的边界不稳定，建议重命名为基础设施语义：

- `infrastructure/http/`
  从 [`src/app/main/lib/http.js`](../../src/app/main/lib/http.js) 迁移基础 HTTP 客户端。
- `infrastructure/tcp/`
  HTTP 传输实现现位于 [`src/app/main/lib/http-client/transport-request.js`](../../src/app/main/lib/http-client/transport-request.js)。
  - （旧 tcp split 文件已移除）
  - `client.js`
  - `heartbeat.js`
  - `fallback-http.js`
  - `platform-sync.js`
  - `target-sync.js`
- `infrastructure/persistence/`
  从 `license-store`、`account-storage`、`session-storage` 收敛本地文件读写规则。
- `infrastructure/electron/`
  收敛窗口创建、菜单、下载、快捷键、缩放、上下文菜单等 Electron 相关适配。

原则很简单：

- 基础设施不关心页面文案和业务按钮状态。
- 领域服务不直接操作 `fs`/`BrowserWindow`/`ipcMain`，而是通过基础设施适配器。

### 8.4 IPC 建议按领域重组

当前 IPC 已经拆目录，但建议继续从“通道集合”升级成“领域门面”：

- `ipc/license.js`
- `ipc/account.js`
- `ipc/network.js`
- `ipc/proxy.js`
- `ipc/browser.js`
- `ipc/system.js`

每个 IPC 文件只做三件事：

- 校验参数。
- 调用对应领域服务。
- 统一返回结果结构。

不要在 IPC 里直接写：

- 文件读写
- Electron 窗口细节
- TCP 协议拼装
- 业务状态拼接

否则后续你会出现“改一个按钮，牵动三个 IPC 文件和两个 service”的问题。

### 8.9 为了稳定性，必须补的工程约束

如果只改目录、不加约束，过两周还是会重新长乱。建议加这几条硬规则：

- `bootstrap` / `composition` 文件不允许出现业务规则判断。
- IPC 文件不允许直接读写本地文件。
- 领域服务不允许直接引用 renderer 术语或 DOM 名称。
- `runtime/app-state.js` 之外不再新增主进程全局可变状态。
- 新增功能必须先选择归属领域，不能先随手放进 `utils/`。
- 超过 500 行的文件默认需要拆分，不接受继续累积。

### 8.10 可以直接执行的最小版本

如果你不想一次改太多，先做最小版本也能明显降成本：

- 新建 `docs/architecture`、`docs/api`、`scripts/windows`、`config`
- 迁移根目录文档和 bat 脚本
- `bootstrap.js` 继续拆向 `composition/startup-guards.js`、`composition/create-ui-bridge.js` 和后续领域服务
- HTTP 客户端已独立为 `http-client.js`
- `auth-cookie.js` 拆成 `auth-service.js` + `cookie-service.js` + `region-parser.js`
- IPC 改成 `license/account/network/proxy/system`

只做这一步，后续维护成本就会明显下降，因为最容易失控的几个入口已经被卡住。
